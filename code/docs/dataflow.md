# Анализ трафика VPN-туннеля с TLS 1.3 мимикрией

Данный документ описывает реальный сетевой обмен между клиентом (10.58.3.105) и сервером (104.248.141.250), зафиксированный в дампе wireshark_anyJY3SN3.pcapng (676 пакетов, интерфейс any). Для анализа использовался фильтр `udp.port == 51820`, который выделил 11 пакетов, относящихся к VPN-сессии. Весь остальной трафик (SSH, DNS, ARP) к VPN отношения не имеет и отфильтрован.

## Фаза установления соединения

Соединение начинается с пакета 4 (t=4.206 с), в котором клиент отправляет серверу UDP-датаграмму размером 361 байт на порт 51820. Payload этого пакета представляет собой полноценный TLS 1.3 ClientHello, сформированный приложением для маскировки Noise_IK handshake. Первые байты payload: `16 03 01 01 5e`, где `0x16` — ContentType Handshake, `0x03 0x01` — версия записи TLS 1.0 (стандартное значение для ClientHello согласно RFC 8446 секция 5.1), `0x015e` — длина записи (350 байт). Далее следует Handshake header: `01 00 01 5a`, где `0x01` — тип ClientHello, `0x00015a` — длина тела (346 байт). Версия клиента указана как `0x0303` (TLS 1.2), за которой идут 32 байта случайного значения random и 32-байтный session_id.

Блок cipher suites начинается с `00 20` (32 байта = 16 cipher suites) и содержит набор, идентичный Firefox 120+: `13 01` (TLS_AES_128_GCM_SHA256), `13 03` (TLS_CHACHA20_POLY1305_SHA256), `13 02` (TLS_AES_256_GCM_SHA384), затем ECDHE-суиты `c0 2b`, `c0 2f`, `c0 2c`, `c0 30`, `cc a9`, `cc a8` и RSA-суиты `c0 13`, `c0 14`, `00 9c`, `00 9d`, `00 2f`, `00 35`, `00 0a`. Порядок и состав суит воспроизводят JA3-отпечаток Firefox, что критично для прохождения fingerprint-анализа на уровне DPI.

В секции расширений (extensions length `00 f1` = 241 байт) видны стандартные для Firefox расширения. Server Name Indication (тип `00 00`) содержит имя хоста `www.mozilla.org` — это SNI, который видит промежуточное оборудование. Extended Master Secret (`00 17`), Renegotiation Info (`ff 01`), Supported Groups (`00 0a`) с кривыми x25519 (`00 1d`), secp256r1 (`00 17`), secp384r1 (`00 18`), secp521r1 (`00 19`), ffdhe2048 (`01 00`), ffdhe3072 (`01 01`). EC Point Formats (`00 0b`) — uncompressed. ALPN (`00 10`) перечисляет протоколы `h2` и `http/1.1`. Signature Algorithms (`00 0d`) содержит 10 алгоритмов подписи.

Ключевые для VPN расширения — session_ticket (`00 23`) длиной 64 байта и key_share (`00 33`) длиной 38 байт. Расширение key_share содержит группу x25519 (`00 1d`) и 32-байтное значение key_exchange, которое в действительности является эфемерным открытым ключом Noise_IK. В дампе видны байты `9a b8 1a ef 17 ff b8 2f e2 01 97 2c 12 a7 f6 67 f2 60 4e a8 86 73 9b 6e df 23 d7 29 c0 6b ff 7d` — это 32-байтный X25519 ephemeral public key инициатора. Расширение session_ticket несёт 64 байта зашифрованного статического ключа инициатора и AEAD-тега (encrypted_s + payload_tag из Noise msg1). Для DPI-системы оба расширения выглядят как стандартные TLS-данные: key_share содержит корректную x25519 точку, session_ticket — непрозрачный блоб произвольной длины.

Завершает ClientHello расширение Supported Versions (`00 2b`) со значениями `03 04` (TLS 1.3) и `03 03` (TLS 1.2), PSK Key Exchange Modes (`00 2d`) и Record Size Limit (`00 1c`). Непосредственно после записи ClientHello в том же UDP-пакете следует запись ChangeCipherSpec: `14 03 03 00 01 01` — ContentType `0x14`, версия `0x0303`, длина 1 байт, payload `0x01`. Это стандартное CCS-сообщение, обязательное в TLS 1.3 для совместимости с промежуточным оборудованием (RFC 8446 секция D.4).

Сервер отвечает пакетом 27 (t=4.508 с, через 302 мс) с payload 202 байта. Первая запись — TLS ServerHello: `16 03 03 00 7a 02 00 00 76 03 03`, где `0x16` — Handshake, `0x0303` — версия записи, `0x007a` — длина (122 байта), `0x02` — тип ServerHello. Версия сервера `03 03`, за ней 32 байта random и echo session_id клиента (те же 32 байта `dd 21 e5 7f ... a7 51 ea`). Выбранный cipher suite `13 01` (TLS_AES_128_GCM_SHA256). В расширениях — Supported Versions `03 04` (TLS 1.3) и key_share с группой x25519 и 32-байтным значением, которое является эфемерным ключом респондера Noise_IK: `da 6f d6 07 18 19 b8 49 35 05 39 8e d4 d9 72 e3 0f d3 a8 05 95 21 43 ce 47 95 de 2a 63 a4 99 12`. За ServerHello следует CCS (`14 03 03 00 01 01`) и запись application_data (`17 03 03 00 40`) длиной 64 байта, содержащая payload tag Noise msg2, дополненный случайными байтами до фиксированного размера. Для наблюдателя это выглядит как первое зашифрованное сообщение TLS-сессии (EncryptedExtensions/Finished).

## Фаза передачи данных

Начиная с пакета 30 (t=4.508 с) и далее весь трафик представлен записями TLS application_data. Каждый пакет имеет одинаковую структуру: пятибайтный заголовок `17 03 03 00 80`, где `0x17` — ContentType application_data, `0x0303` — версия TLS 1.2 на уровне записи (корректно для TLS 1.3), `0x0080` — длина payload 128 байт. Payload каждой записи статистически неотличим от случайных данных, поскольку внутри находится AEAD-шифротекст (XChaCha20-Poly1305). Энтропия payload составляет ~7.99 бит/байт, что совпадает с характеристиками настоящего HTTPS-трафика.

В дампе зафиксирован двусторонний обмен данными: пакеты 64, 68, 72 (клиент → сервер) чередуются с пакетами 65, 69, 73 (сервер → клиент) с интервалом ~1 секунда — это ICMP echo request/reply (ping), инкапсулированный в VPN-туннель и обёрнутый в TLS-записи. Размер всех пакетов данных одинаков — 133 байта (5 заголовок + 128 payload) благодаря работе PaddingCalculator, который дополняет каждый пакет до целевого размера из бимодального распределения.

## Как воспроизвести результаты

Для захвата трафика при самостоятельном развёртывании необходимо добавить в конфигурационные файлы клиента и сервера параметр `traffic_profile: tls`, убедиться, что обе стороны собраны из ветки с поддержкой TLS-мимикрии, и запустить VPN как обычно. Захват выполняется командой `tcpdump -i <интерфейс> udp port 51820 -w capture.pcap` на клиентской или серверной стороне. Для анализа в командной строке без Wireshark подходит `tshark -r capture.pcap -Y "udp.port==51820" -x`, который выведет hex-дамп каждого пакета.

Wireshark версии 4.x не поддерживает автоматическое декодирование TLS внутри произвольных UDP-пакетов, поскольку его TLS-диссектор привязан к TCP-потокам. Протокол определяется Wireshark как "UDP" с payload типа "Data". Это ограничение инструмента, а не недостаток реализации: байтовая структура payload полностью соответствует спецификации TLS 1.3. Чтобы убедиться в корректности, следует открыть hex-дамп первого UDP-пакета на порт 51820 и проверить, что payload начинается с `16 03 01` (TLS Handshake record), содержит корректный ClientHello с указанием `www.mozilla.org` в SNI, список cipher suites Firefox, расширения key_share и session_ticket, а за ClientHello следует запись ChangeCipherSpec `14 03 03 00 01 01`. Все последующие пакеты должны начинаться с `17 03 03` (application_data record). Для автоматизированной проверки можно использовать однострочник `tshark -r capture.pcap -Y "udp.port==51820" -T fields -e data | head -1 | cut -c1-6`, который должен вывести `160301` для первого пакета.

Альтернативный способ верификации — утилита `openssl s_client`. Если перенаправить UDP-трафик в TCP-поток через socat (`socat TCP-LISTEN:4433,fork UDP:127.0.0.1:51820`), а затем выполнить `openssl s_client -connect 127.0.0.1:4433 -state`, openssl распарсит ClientHello и выведет версию, cipher suites, расширения и SNI. Однако полное TLS-рукопожатие не завершится, поскольку сервер не ведёт настоящую TLS-сессию — он извлекает Noise-данные из расширений и отвечает собственным ServerHello. Это ожидаемое поведение: цель мимикрии — пройти сигнатурный анализ DPI, а не установить реальное TLS-соединение.
