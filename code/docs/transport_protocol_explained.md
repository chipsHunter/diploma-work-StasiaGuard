# Обоснование выбора UDP в качестве транспортного протокола

## Проблема TCP-over-TCP

VPN-туннель по своей природе инкапсулирует пользовательский трафик, который в подавляющем большинстве случаев уже содержит TCP-соединения: HTTP-запросы, SSH-сессии, почтовые протоколы. Если транспортом самого туннеля тоже является TCP, формируется двухуровневая конструкция, в которой TCP-соединение пользователя вложено в TCP-соединение туннеля.

Каждый из двух уровней TCP обладает собственным механизмом контроля перегрузки (congestion control) и повторной передачи (retransmission). Когда внешний TCP-уровень обнаруживает потерю пакета, он инициирует ретрансмит и снижает окно перегрузки. Одновременно с этим внутренний TCP фиксирует увеличение RTT (из-за ожидания ретрансмита на внешнем уровне) и тоже начинает снижать скорость передачи. Два независимых алгоритма congestion control вступают в конфликт: каждый пытается адаптироваться к условиям сети, но реагирует не на реальные потери, а на артефакты работы другого уровня. Этот эффект известен в литературе как TCP meltdown. На практике при уровне потерь 2-3% пропускная способность TCP-over-TCP может падать на порядок по сравнению с TCP-over-UDP, поскольку фаза восстановления занимает в разы больше времени из-за взаимного подавления окон.

## Head-of-line blocking

TCP гарантирует строгий порядок доставки данных: если пакет с порядковым номером N потерян, все последующие пакеты (N+1, N+2, ...), даже если они уже получены приёмной стороной, будут удерживаться в буфере до тех пор, пока потерянный пакет не будет доставлен повторно. В контексте VPN это означает, что потеря одного транспортного пакета блокирует доставку всех последующих VPN-пакетов, даже если они принадлежат совершенно разным пользовательским соединениям. Пользовательская SSH-сессия будет ждать, пока не будет ретранслирован пакет, содержащий фрагмент совершенно другого HTTP-запроса.

UDP не имеет этой проблемы. Каждая датаграмма обрабатывается независимо. Потеря одного VPN-пакета приводит к потере ровно одного инкапсулированного пользовательского пакета, после чего пользовательский TCP самостоятельно выполнит ретрансмит на своём уровне. Остальные соединения не затрагиваются. Такой подход обеспечивает минимальную задержку и максимальную утилизацию канала.

## Совместимость с Noise_IK

Протокол рукопожатия Noise_IK, используемый в данной реализации, завершается за один round-trip (два сообщения: msg1 от инициатора и msg2 от респондера). Для UDP этого достаточно: клиент отправляет датаграмму с msg1, сервер отвечает датаграммой с msg2. В случае потери клиент повторяет отправку по простому таймауту (5 секунд, 3 попытки). Накладные расходы на установление соединения TCP (3-way handshake + TLS handshake) были бы избыточны, учитывая что Noise_IK сам обеспечивает полную аутентификацию и согласование ключей.

## Соответствие отраслевой практике

Выбор UDP в качестве транспорта VPN-туннеля является стандартом де-факто в индустрии. WireGuard использует исключительно UDP. OpenVPN рекомендует UDP как основной транспорт и поддерживает TCP только как fallback для сетей с жёсткой фильтрацией. Протокол QUIC (HTTP/3), принятый IETF в RFC 9000, также построен поверх UDP с интеграцией TLS 1.3 непосредственно в транспортный уровень.

Последнее обстоятельство особенно значимо для модуля маскировки трафика: TLS-подобные записи поверх UDP являются нормой в современных сетях благодаря повсеместному распространению QUIC. DPI-системы, настроенные на анализ TLS, ожидают встретить TLS-записи не только в TCP-потоках, но и в UDP-датаграммах. Это делает наш подход (TLS 1.3 record layer поверх UDP) структурно близким к легитимному QUIC-трафику и снижает вероятность ложного срабатывания фильтров.
